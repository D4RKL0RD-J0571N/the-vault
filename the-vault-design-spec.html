01 — Landscape

What Already Exists

The tooling space has converged around active development assistance — tools that help you inside the IDE you're currently working in. Nobody has built a tool oriented around personal code archaeology — navigating your own history across years and dozens of abandoned projects.

Tool Multi-Project Local / Private Symbol Docs History / Timeline Abandonment Analysis
Cursor ✗ Single project ~ Cloud index ~ Hover only ✗ ✗
GitHub Copilot ✗ Current repo ✗ Cloud ~ Inline only ✗ ✗
RooCode + Qdrant ✗ Single project ✓ Local option ✗ ✗ ✗
Potpie ✗ Per-repo ✗ Cloud-first ~ Knowledge graph ✗ ✗
AnythingLLM ~ Workspaces ✓ Desktop app ✗ Not code-aware ✗ ✗
Sourcegraph Cody ~ Enterprise ✗ Cloud ✓ ✗ ✗
The Vault ✓ Core feature ✓ 100% local ✓ Stored, not live ✓ Core feature ✓ Core feature
The fundamental difference: every existing tool is designed to help you write code now. The Vault is designed to help you find and understand code you wrote then. This is a completely different use case with no real competition.
02 — Architecture

System Design

The architecture is built around a single guiding principle: pay the AI cost once, get the value forever. Everything is designed to minimize redundant computation while maximizing the depth of stored knowledge.

Two-Layer Processing Philosophy

The parser layer (Tree-sitter) is fast, deterministic, and free — it runs in milliseconds. The AI layer (Ollama) is slow and expensive in compute time, but runs only once per file version and produces stored knowledge. These two layers should never be confused or combined.

Tree-sitter → Symbol Extraction
Ollama → Documentation Generation
Qdrant → Semantic Search
SQLite → Relational Metadata
JSON Cache → Portable Doc Store

04 — Features

Feature Specification

Must Have — The App Lives or Dies By These

Must Have

Project Registry

Grid of project cards. Each shows name, type, last modified, LOC, one-line AI summary, and a health color indicator (green / amber / red). Filterable by language, type, recency.

Must Have

Background Crawler

Watches configured root directories. Detects new files and changed files. Queues them for parsing and AI documentation. Shows live progress. Runs silently after initial full scan.

Must Have

Symbol-Level AI Docs

Every class, method, field, enum, interface documented by Ollama — once, stored permanently. Summary, purpose, pattern detected, reuse rating, and usage example.

Must Have

Semantic Search

"Find my XP scaling formula" → returns BaseSkill.cs from Project-X. Natural language queries via Qdrant vector search. Results show project, file, symbol name, and AI summary.

Must Have

Project Detail View

File tree, project-level AI summary, detected patterns, key systems, all symbols with documentation. Entry point to the whole project knowledge base.

Must Have

Code Viewer + Docs Panel

Open any file with syntax highlighting. Click a class or method to open its AI documentation panel on the right. Like an IDE tooltip but much richer and always available.

Nice to Have — Make It Genuinely Great

Nice to Have

Development Timeline

Horizontal timeline built from file timestamps. Scrub through your history. See which projects were active when. Understand your context: "I was doing HiperBanco while starting Unity."

Nice to Have

Cross-Project Pattern Map

Visual graph showing how the EventBus evolved across SnakePrototype → Project-X → VisualNovelEngine. Links similar systems across projects. Your technical DNA, visualized.

Nice to Have

TODO/FIXME Tracker

Global list of every TODO and FIXME across all projects. Grouped by project. AI's best guess at what was intended. Effort estimate to complete. A window into your unfinished thoughts.

Nice to Have

Code Snippet Pinboard

Pin specific code snippets from any project. Source file, project, and AI docs visible per pin. Create collections: "Pooling Patterns", "Authentication Logic", "Math Utilities".

Nice to Have

Abandonment Diagnosis

Per-project "Why did I stop?" analysis. AI examines last modified files, WIP comments, git history. Categorizes: TechnicalWall / ScopeCreep / ContextSwitch / NaturalCompletion.

Nice to Have

Git History Integration

Pull commit logs for git repos. Enrich project timelines with real commit messages. Last commit subject, branch state, frequency over time. Real data beats timestamp guessing.

Nice to Have · Phase 5

Local Knowledge Graph Export

Export .vault/graph.json mapping relationships between systems across all projects — e.g. "EventBus pattern → implemented in 3 games, evolved across 2 years." Built on top of pattern detection data already in storage. YARVIS ingests this as a static knowledge artifact to synthesize architecture recommendations without querying the live database.

Nice to Have · Phase 5

Refactor Potential Scoring

After the initial documentation pass, a second-pass Ollama prompt groups all symbols sharing the same detected pattern and ranks them by code quality, recency, and reuse potential. The winner is flagged as a Refactor Seed — the definitive version of that system, ready for extraction as a standalone package or Unity Asset Store candidate. A dedicated "Seeds" filter in the Registry surfaces these instantly.

Dream Features — If This Becomes Something Bigger

Dream

Per-Project AI Chat

Bounded RAG chat per project. "How does attribute regeneration work?" — answered using only that project's indexed symbols as context. No hallucination about things that aren't there.

Dream

Asset Package Assembler

Highlight a system (e.g. Typed Event Bus). App finds all dependencies, copies them to a clean folder, generates a README. One-click extraction of reusable modules.

Dream

"Port This To…" Button

Highlight a class, choose a target language. Ollama generates a translation with explanatory comments. Immediately useful for your Java → C# patterns.

Dream

YARVIS Feed

Dedicated export pipeline: symbols, summaries, project graph, cross-references → structured format YARVIS can query directly. The Vault becomes YARVIS's long-term code memory. Pairs with the CLI Recall command below.

Dream · Depends on YARVIS Plugin System

YARVIS CLI Recall

Once The Vault's REST API is stable, YARVIS gains a recall command that queries it with natural language. yarvis recall "best inventory system I ever wrote" fires a semantic search, retrieves the top-ranked symbol docs, and formats them into a YARVIS response. Optionally chain with --port-to cs to immediately translate the result. Architecturally simple — just an HTTP call to POST /api/search. You'd literally be talking to your past self.

A .vault/ hidden folder is created inside each indexed project root. It stores symbols/*.json (AI docs per symbol), project.json (metadata snapshot), and at the global level graph.json (cross-project knowledge graph) and seeds.json (Refactor Seed rankings). All of these travel with the project if you move it, and YARVIS can consume any of them without touching the live database.
06 — AI Integration

Ollama Prompt Design

Prompt quality determines documentation quality. Every prompt is a template with the symbol's raw code, type, context (parent class, file path), and language injected.

## SYMBOL DOCUMENTATION PROMPT TEMPLATE You are a senior software engineer analyzing legacy code. Your task is to generate structured documentation for a single code symbol. Language: {language} Symbol Type: {symbol_type} File: {file_path} Parent Context: {parent_class_name} (if applicable) Project: {project_name} ({project_type}) CODE: ```{language} {raw_code} ``` Respond ONLY with valid JSON. No preamble. Format: { "summary": "One sentence. What this does.", "purpose": "Why it exists. Design intent.", "pattern": "Design pattern if detected, else null", "reuse": "HIGH | MEDIUM | LOW | PROJECT_SPECIFIC", "example": "One usage example as a code string, else null", "params": [{"name":"", "type":"", "desc":""}], "returns": "What is returned, or null", "notes": "Caveats, gotchas, tech debt observations, or null" }

Keep prompts tight. A bloated prompt wastes tokens, slows generation, and reduces JSON reliability. The structure forces the model to be concise.

07 — Build Plan

Development Phases

Each phase delivers something usable. You're never building in the dark.

01

The Scanner — No AI, Just Data

Build the crawler and Tree-sitter parser. Get a working SQLite database of all your projects and their symbols with zero AI involved. At the end of this phase you have a full symbol table. Validate it against your known projects. Deliverable: CLI tool that prints a project/symbol report.

02

The Registry — Frontend with Static Data

Build the Tauri + React frontend. Wire it to the FastAPI backend. Show the project registry grid with real project metadata. No AI docs yet — just file counts, LOC, last modified. Deliverable: Desktop app showing your projects as cards.

03

The Intelligence — Ollama Documentation

Wire in the Ollama queue. Run the first full documentation pass as a background job. Store results in SQLite + JSON cache. Show AI docs in the Code Viewer panel. Deliverable: Working symbol documentation for all parsed symbols.

04

The Search — Qdrant + Semantic Queries

Set up Qdrant (Docker). Embed all symbol summaries using nomic-embed-text via Ollama. Build the search API and frontend. Test with real queries from your history. Deliverable: Semantic search across all symbols.

05

Polish + Nice-to-Haves

Git history integration, Timeline view, TODO tracker, Abandonment diagnosis, Pinboard. These can be built independently and added without breaking anything. Deliverable: Fully featured v1.0.
