01 — Foundation

Repository Structure

The root of the repo communicates the entire project to anyone — human or AI — in under ten seconds. Every file at the root level should have a clear, unambiguous purpose. Nothing accidental should live here.

the-vault/ # Root — mono-repo, everything lives here │ ├── .github/ # GitHub metadata │ ├── ISSUE_TEMPLATE/ │ │ ├── bug_report.md │ │ └── feature_request.md │ ├── workflows/ │ │ ├── ci.yml # Lint + test on push │ │ └── release.yml # Tag → GitHub Release + changelog │ └── PULL_REQUEST_TEMPLATE.md │ ├── backend/ # FastAPI (Python) ├── frontend/ # Tauri + React (TypeScript) ├── docs/ # All human-readable documentation │ ├── architecture.md # System design narrative │ ├── data-models.md # All models with field definitions │ ├── api-reference.md # All API endpoints │ ├── ai-prompts.md # All Ollama prompt templates │ ├── decisions/ # ADRs — Architecture Decision Records │ │ ├── 001-tauri-vs-electron.md │ │ ├── 002-qdrant-vs-chroma.md │ │ └── 003-tree-sitter-parsers.md │ └── setup.md # Dev environment setup guide │ ├── AGENTS.md # ← AI assistant master context file ├── .cursorrules # ← Cursor-specific rules ├── .windsurfrules # ← Windsurf-specific rules ├── .antigravity # ← Antigravity config (if applicable) │ ├── README.md # Project overview — first thing anyone reads ├── CHANGELOG.md # Auto-generated from commits ├── CONTRIBUTING.md # How to contribute (even if it's just you) ├── docker-compose.yml # Qdrant container └── .gitignore
KEY PRINCIPLE The highlighted files — AGENTS.md, .cursorrules, .windsurfrules — are not documentation. They are active configuration that shapes how your AI tools behave in every session. They are as important as your pyproject.toml.
02 — Git Workflow

Branching Strategy

For a solo project with phased delivery, a simplified GitHub Flow is the right choice. Gitflow is overkill here — it adds ceremony without benefit when there's one developer. The model is simple: main is always shippable. Everything else is a branch.

BranchPurposeProtectionMerges Into
main Production-ready at all times. Tagged releases live here. ✓ Protected · No direct push · PR required —
dev Integration branch. Working but not necessarily polished. Where phases land before being promoted. ✓ Protected · PR required main (via PR + tag)
feat/scanner Feature branches. One per feature/task. Short-lived — merge and delete. — dev
fix/qdrant-timeout Bug fix branches. Same pattern as features. — dev (or main for hotfixes)
chore/update-deps Non-feature work: dependency updates, config tweaks, refactors. — dev
docs/architecture-adr Documentation-only changes. Can go straight to main via PR. — main or dev

Branch Naming Convention

Branch name patterns
# Format: <type>/<short-kebab-description> feat/project-scanner # Phase 1 — the file crawler feat/tree-sitter-parser-cs # C# symbol extractor feat/ollama-doc-queue # Phase 3 — AI documentation generation feat/qdrant-semantic-search # Phase 4 — vector search feat/registry-ui # Frontend project grid fix/watcher-missed-renames # Bugfix for watchdog rename events fix/ollama-json-parse-failure # AI response parsing edge case chore/bump-fastapi-0.115 # Dependency update chore/refactor-symbol-models # Internal cleanup docs/adr-003-parser-choice # Architecture Decision Record
RULE Never commit directly to main or dev. Even working alone, the PR process forces you to review your own diff before merging — this catches more mistakes than any linter. AI assistants also produce better code when they know a review step is expected.
03 — Commit Convention

The Language of History

Conventional Commits is the standard, and it matters here for two reasons: it makes CHANGELOG.md auto-generatable, and it gives your AI assistants context about what changed and why when they read your git log.

Commit message format
<type>(<scope>): <short imperative description> # Optional body — explain WHY, not what (the diff shows what) [blank line] [body — wrapped at 72 chars] # Optional footer — breaking changes, issue refs [blank line] [BREAKING CHANGE: description] [Closes #42]

Types

TypeWhen to UseAppears in Changelog?
featNew feature or capability✓ Yes — Features
fixBug fix✓ Yes — Bug Fixes
docsDocumentation changes onlyNo
choreBuild, tooling, deps, configNo
refactorCode restructure, no behavior changeNo
perfPerformance improvement✓ Yes — Performance
testAdding or fixing testsNo
ciCI/CD pipeline changesNo

Scopes for The Vault

Scopes are the part in parentheses — they narrow which subsystem changed. Agree on these upfront so they're consistent.

Commit examples with scopes
feat(crawler): add recursive directory scanner with exclusion rules feat(parser): implement C# symbol extractor via tree-sitter feat(parser): add Java parser for class and method symbols feat(ai): wire Ollama doc generation queue with retry logic feat(search): integrate Qdrant for semantic symbol search feat(ui): add project registry grid with health color indicators feat(ui): implement code viewer with AI docs panel feat(graph): export .vault/graph.json knowledge graph feat(api): add GET /refactor/seeds endpoint fix(crawler): handle symlinks without infinite loop fix(ai): gracefully handle ollama JSON parse failure fix(parser): skip binary files misidentified as source chore(deps): bump qdrant-client to 1.9.0 refactor(storage): split database.py into models and session modules docs(adr): add decision record for tree-sitter vs regex parsing # Breaking change example feat(api)!: rename /index to /indexer for consistency BREAKING CHANGE: all clients must update endpoint from /index to /indexer Closes #18
04 — Versioning & Tagging

Marking Milestones

Semantic Versioning (semver) tells anyone reading the tag exactly what changed. For The Vault, tags also correspond directly to completed build phases — so the history is human-readable even years later.

SemVer Rules

Version PartWhen It BumpsExample
MAJOR (x.0.0)Breaking change — existing .vault/ files or APIs incompatible2.0.0
MINOR (0.x.0)New feature added, backward compatible1.3.0
PATCH (0.0.x)Bug fix, no new capability1.3.1

Phase-to-Tag Mapping

v0.1.0-alpha Phase 1 complete — scanner + symbol table
v0.2.0-alpha Phase 2 complete — frontend registry, no AI yet
v0.3.0-beta Phase 3 complete — Ollama docs live
v0.4.0-beta Phase 4 complete — semantic search working
v1.0.0 Phase 5 complete — full featured, stable
v1.1.0 Knowledge Graph Export added
v1.2.0 Refactor Seed Scoring added
Git tag commands
# Always tag from main, never from dev git checkout main git pull origin main # Create annotated tag (not lightweight — annotated has a message) git tag -a v0.1.0-alpha -m "Phase 1: Scanner and symbol extractor complete" # Push the tag to GitHub (triggers release workflow) git push origin v0.1.0-alpha # List all tags git tag -l --sort=-v:refname
05 — AI Assistant Configuration

Teaching Your Tools

This is the most important section if you're using Cursor, Windsurf, or Antigravity. These files are read at the start of every AI session and shape every suggestion the tool makes. A well-written config means you never have to re-explain the architecture. A missing or bad config means the AI invents things.

STRATEGY Write AGENTS.md first as the single source of truth — the full project context, architecture, conventions, and rules. Then each tool-specific file (.cursorrules, .windsurfrules) references AGENTS.md and adds only tool-specific overrides. You maintain one document, not three.

AGENTS.md — The Master Context File

This is read by any AI agent that follows the emerging AGENTS.md convention (supported by Codex, many Cursor setups, and increasingly by other tools). It's also what you paste into a new chat when starting a session in Claude or any other AI.

AGENTS.md / (repo root)
# The Vault — Agent Context ## Project Identity The Vault is a local-first, AI-powered code archaeology desktop application. It indexes personal development projects across multiple drives, generates AI documentation for every code symbol (class, method, enum, etc.) using a local Ollama instance, and provides semantic search across the entire history. ## Tech Stack - Backend: FastAPI (Python 3.11+), SQLAlchemy, SQLite - AI: Ollama (WSL2, RTX 3060 12GB) — Qwen2.5-Coder 7B Instruct - Vectors: Qdrant (Docker) + nomic-embed-text for embeddings - Parser: tree-sitter (Python bindings) — C#, Java, Python, JS, RenPy - Frontend: Tauri v2 + React 18 + TypeScript 5 + Tailwind CSS - Storage: SQLite (metadata), Qdrant (vectors), .vault/*.json (AI doc cache) ## Repository Structure - backend/vault/ FastAPI application - backend/vault/crawler/ File watcher and project scanner - backend/vault/parser/ Tree-sitter symbol extraction - backend/vault/ai/ Ollama client, doc generation queue - backend/vault/storage/ SQLAlchemy models, Qdrant wrapper, JSON cache - backend/vault/api/ FastAPI route handlers - frontend/src/pages/ Tauri/React page components - frontend/src/components/ Reusable UI components - docs/ Architecture docs, ADRs, API reference ## Architecture Rules — Never Violate These 1. The parser layer (tree-sitter) NEVER calls Ollama. These are separate layers. 2. Ollama is called ONLY from backend/vault/ai/ — nowhere else. 3. All file system access on the user's drives goes through the crawler module. 4. The .vault/ hidden folder in each project is append-only. Never delete user files. 5. All Qdrant operations go through storage/vector_store.py — never call Qdrant directly. 6. FastAPI routes are thin — business logic lives in the service layer, not in routes. 7. The frontend NEVER reads files directly — all data goes through the FastAPI API. ## Coding Conventions ### Python (backend) - Use type hints everywhere. No `Any` without a comment explaining why. - Async/await for all I/O — never blocking calls in async functions. - Pydantic v2 for all data validation and API schemas. - Error handling: use custom exception classes from vault/exceptions.py. - Logging: use structured logging via `structlog`, never print(). - Tests: pytest, aim for coverage on parser and AI modules especially. ### TypeScript (frontend) - Strict mode enabled. No `any` without comment. - All API calls go through lib/api.ts — never raw fetch() in components. - Component props must be typed with interfaces, not inline types. - No business logic in components — hooks only. ## Current Build Phase # UPDATE THIS EVERY TIME YOU START A NEW PHASE Phase: 1 — Scanner and Parser (no AI, no frontend yet) Active branch: feat/project-scanner Focus: backend/vault/crawler/ and backend/vault/parser/ ## Known Constraints - Ollama runs in WSL2. The API endpoint is http://localhost:11434 - Qdrant runs in Docker on port 6333. - Never suggest cloud APIs or external services — this is a 100% local tool. - RTX 3060 has 12GB VRAM. Recommended model: Qwen2.5-Coder:7b-instruct-q4_K_M ## Design Decisions (see docs/decisions/ for full ADRs) - Tauri over Electron: smaller binary, native file system access, Rust shell - Qdrant over ChromaDB: better performance, Docker-based, production-ready - Tree-sitter over regex: grammar-aware, handles nested structures, multi-language - SQLite over PostgreSQL: zero-config, file-based, perfect for local-first tools

.cursorrules — Cursor-Specific Rules

.cursorrules / (repo root)
# Cursor Rules for The Vault # Full project context is in AGENTS.md — read that first. ## Behavior - Always read AGENTS.md before suggesting architectural changes. - When editing backend files, check that the change respects the layer separation in AGENTS.md. - When asked to add a feature, ask which phase it belongs to before writing code. - Prefer editing existing files over creating new ones unless a new module is clearly needed. - Always suggest tests alongside new functions. ## Code Style — Python - Maximum line length: 100 characters. - Imports: stdlib, then third-party, then local — separated by blank lines. - Use `from __future__ import annotations` at top of every Python file. - Docstrings: Google style, required for all public functions and classes. ## Code Style — TypeScript - Use named exports, never default exports (except for page components). - Tailwind classes: utility-first, no custom CSS unless Tailwind cannot do it. ## Forbidden - Do NOT suggest installing cloud SDKs (OpenAI, Anthropic, etc.) - Do NOT use subprocess or shell=True in Python code. - Do NOT use localStorage in the frontend — Tauri has its own storage API. - Do NOT add dependencies without explaining why the existing stack cannot handle it. ## When Unsure Ask. Do not guess at architecture decisions. Refer to docs/decisions/ ADRs.

.windsurfrules — Windsurf-Specific Rules

.windsurfrules / (repo root)
# Windsurf Rules for The Vault # Full context: AGENTS.md. Architecture decisions: docs/decisions/ ## Cascade Behavior - Before any multi-file edit, state which files will be touched and why. - Respect the layer architecture from AGENTS.md. Parser never calls AI. Routes are thin. - When generating new API endpoints, always generate the corresponding Pydantic schema. - When editing the frontend, always generate the corresponding TypeScript type in lib/types.ts. ## Flow Rules - Complete one logical change at a time. Do not refactor while adding features. - If a change requires touching more than 5 files, pause and describe the plan first. - Always run the affected test suite mentally before suggesting the change. ## Style Same as .cursorrules — Google docstrings, Pydantic v2, typed hooks, named exports. ## Forbidden Same as .cursorrules. No cloud APIs. No subprocess shell=True. No localStorage.
Antigravity and other tools typically look for AGENTS.md or a tool-specific config file. Check the tool's documentation for the exact filename. The content strategy is the same: point to AGENTS.md for the full context, add only tool-specific behavior rules in the tool file.
06 — Docs as Infrastructure

Documentation Files

Documentation in a project like this serves two audiences simultaneously: you (the human returning after two weeks away) and the AI assistant (starting fresh every session). Writing for both is the same act — be precise, be explicit, assume nothing is obvious.

Architecture Decision Records (ADRs)

Every time you make a significant architectural choice — choosing Qdrant over ChromaDB, Tauri over Electron, tree-sitter over regex — write an ADR. These prevent you and your AI tools from revisiting the same decisions repeatedly. The format is intentionally brief.

docs/decisions/002-qdrant-vs-chroma.md — ADR template
# ADR-002: Qdrant over ChromaDB for Vector Storage ## Status Accepted — 2026-03-01 ## Context The Vault needs a local vector store for semantic search across code symbol embeddings. Two main candidates: Qdrant and ChromaDB. Both run locally. ## Decision Use Qdrant, deployed via Docker on port 6333. ## Reasons - Qdrant has a stable REST API that works well from Python and TypeScript. - Better performance at the scale we expect (50k–500k vectors). - Docker-based — no Python version conflicts with the backend environment. - ChromaDB has had API stability issues across minor versions. ## Consequences - Requires Docker to be running for the full app to work. - Added docker-compose.yml to manage the Qdrant container. - All vector operations must go through storage/vector_store.py to allow future backend swap without touching business logic.

The Files and What They're For

README.md

What the project does, why it exists, how to set it up, a screenshot or diagram. The first file anyone reads. Keep it current — an outdated README is worse than none.

AGENTS.md

The complete AI context file. Architecture, rules, current phase, known constraints. Updated at the start of every new phase. The most important file in the repo for AI-assisted development.

CHANGELOG.md

Auto-generated from conventional commits using git-cliff or standard-version. Never written by hand. Updated automatically on each tag. The human-readable history of what changed.

CONTRIBUTING.md

How to contribute — even if it's just you. Documents the branch naming convention, commit format, PR process, and where to put docs. Makes your own rules explicit so you follow them.

docs/architecture.md

The system design narrative. The ASCII architecture diagram from the design spec lives here. Updated when the architecture changes. This is what you paste into a new AI session when AGENTS.md isn't enough.

docs/decisions/*.md

Architecture Decision Records. One file per major decision. The why behind every architectural choice. Prevents relitigating the same decisions when you return or when an AI tool suggests a "better" alternative you already considered.

docs/api-reference.md

All API endpoints documented: path, method, request schema, response schema, example. Kept in sync with the FastAPI code. Can also be auto-generated from FastAPI's OpenAPI output.

docs/ai-prompts.md

All Ollama prompt templates used by the app. Version-controlled because prompts are code — they directly determine output quality and should be treated with the same rigor.

08 — Automation

CI/CD — Lightweight but Real

Two workflows. That's all you need for a solo project. One that runs on every push, one that runs on every tag. Don't over-engineer this.

.github/workflows/ci.yml — Runs on every push and PR
name: CI on: push: branches: [main, dev] pull_request: branches: [main, dev] jobs: backend: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-python@v5 with: python-version: '3.11' - run: pip install -e ".[dev]" working-directory: backend - name: Lint (ruff) run: ruff check . working-directory: backend - name: Type check (mypy) run: mypy vault/ working-directory: backend - name: Tests (pytest) run: pytest tests/ -v working-directory: backend frontend: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 with: node-version: '20' - run: npm ci working-directory: frontend - name: Type check (tsc) run: npx tsc --noEmit working-directory: frontend - name: Lint (eslint) run: npm run lint working-directory: frontend
.github/workflows/release.yml — Runs on every version tag push
name: Release on: push: tags: - 'v*' jobs: release: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 with: fetch-depth: 0 # Full history for changelog generation - name: Generate Changelog (git-cliff) uses: orhun/git-cliff-action@v3 with: config: cliff.toml args: --latest --strip header id: changelog - name: Create GitHub Release uses: softprops/action-gh-release@v2 with: body: ${{ steps.changelog.outputs.content }} draft: false prerelease: ${{ contains(github.ref, 'alpha') || contains(github.ref, 'beta') }}
09 — Getting Started

The First Day Checklist

Do these in order. Each step is a prerequisite for the next.

01

Create the GitHub Repository

Name: the-vault. Private (for now). Initialize with a README. Do not let GitHub create a .gitignore — you'll write a precise one. Clone it locally.

02

Create the Root Structure

Create backend/, frontend/, docs/, docs/decisions/, .github/workflows/, .github/ISSUE_TEMPLATE/. Commit empty .gitkeep files to preserve the directories.

Quick scaffold command
mkdir -p backend frontend docs/decisions .github/workflows .github/ISSUE_TEMPLATE touch .github/workflows/ci.yml .github/workflows/release.yml touch AGENTS.md .cursorrules .windsurfrules CHANGELOG.md CONTRIBUTING.md git add . && git commit -m "chore: initialize repository structure"
03

Write AGENTS.md First

Before touching any code. Use the template from Section 05 of this document. Fill in the Current Build Phase section to say Phase 1 — Scanner and Parser. This is the document that will ground every AI session.

04

Write .cursorrules and .windsurfrules

Use the templates from Section 05. These reference AGENTS.md so they stay light. Commit both files on main before starting feature branches.

05

Configure Branch Protection

In GitHub → Settings → Branches: add protection rules for main and dev. Require pull requests. Enable "Delete branch on merge".

06

Create the dev Branch

git checkout -b dev && git push origin dev. Set it as the default branch for PRs in GitHub Settings.

07

Write ADR-001: The Architecture Decision

Document the core technology choices as docs/decisions/001-core-stack.md. Reference the design specification. This is your "why we built it this way" that prevents future second-guessing.

08

Open Your First Feature Branch

git checkout dev && git checkout -b feat/project-scanner. You are now ready to write Phase 1. Every future work item follows this same branch → PR → merge flow.

09

Update AGENTS.md at Every Phase Boundary

When Phase 1 is complete and merged, update the Current Build Phase section in AGENTS.md before opening the first Phase 2 branch. This single habit ensures your AI tools always have accurate context about where you are in the build.

THE GOLDEN RULE The most expensive mistake in AI-assisted development is starting to code before the governance files exist. An AI tool without AGENTS.md is a very capable stranger who doesn't know your codebase, your conventions, or your decisions. Write the rules first, then open Cursor.
